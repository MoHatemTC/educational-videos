1. SYSTEM OVERVIEW
────────────────────────────────────────────────────────────

The pipeline is structured as three sequential, independently
testable phases:

  Phase 1 — Self-Healing Code Execution
  Phase 2 — Multi-Lingual TTS Synthesis
  Phase 3 — Timeline Stretch & Alignment

All phases are orchestrated by pipeline.py::Pipeline and
communicate via Pydantic v2 typed contracts (no raw dicts).


────────────────────────────────────────────────────────────
2. PHASE 1 — SELF-HEALING CODE EXECUTION
────────────────────────────────────────────────────────────

2.1 Sandbox Runner (sandbox/runner.py)
--------------------------------------
• Docker path (primary): uses docker.from_env(), mounts a
  temp .py file read-only, disables network, caps memory
  (config.docker_mem_limit, default 256 MB), removes the
  container automatically.

• Subprocess fallback (stdlib): spawns sys.executable in a
  subprocess with capture_output=True.  On POSIX, a
  preexec_fn applies resource.setrlimit caps:
    – RLIMIT_CPU:    10 CPU-seconds
    – RLIMIT_FSIZE:  10 MB max file write
    – RLIMIT_NOFILE: 64 open file descriptors
  subprocess.run() enforces a wall-clock timeout
  (config.timeout_seconds, default 15 s).

• Both paths respect config.max_output_bytes to cap
  stdout+stderr before storing.

2.2 Output Parser (sandbox/parser.py)
--------------------------------------
• Splits stderr on "Traceback (most recent call last):" to
  handle multiple chained exceptions.

• For each block, extracts every frame via:
    File "<filename>", line <N>, in <function>
  and stores the associated source line when present.

• Identifies the exception type with a two-pass regex that
  handles dotted names (e.g. module.SubError) and the
  special SyntaxError format.

• first_failing_line is set to the deepest (innermost)
  frame's lineno for prompt annotation.

• TimeoutExpired stderr is detected and given its own
  error_type so the LLM doesn't try to "fix" a timeout.

2.3 Prompt Builder (sandbox/prompt_builder.py)
------------------------------------------------
• build_correction_prompt(result, attempt, history) →
  (system_prompt, user_prompt)

• The broken code is annotated inline:
    seq.appnd(x)  # ← ERROR HERE (line 4)

• The last 3 prior-attempt codes are included in a
  "Prior Corrections" block so the LLM avoids repeating
  failed approaches.

• System prompt enforces: no markdown fences, no new
  external imports, preserve intent, mock missing files,
  output must be complete and runnable.

2.4 Self-Healing Loop (sandbox/loop.py)
-----------------------------------------
• Bounded: 1 initial run + up to max_correction_attempts
  LLM correction rounds (default 5).

• Early-exit if the LLM returns None (API error) or
  identical code (no progress).

• Every attempt is appended to logs/execution_log.jsonl
  as a JSONL record with timestamp, attempt number,
  success flag, errors, stdout preview, and code length.

• Returns a HealingResult with:
    healed         — bool
    final_code     — the last code string
    final_result   — ExecutionResult of the last run
    attempts       — total runs executed
    history        — [(code, result), ...] for audit

• The timeline sync in Phase 3 uses final_code as the
  validated source, NOT the pre-correction draft.

3. PHASE 2 — MULTI-LINGUAL TTS SYNTHESIS
────────────────────────────────────────────────────────────

3.1 TTS Client (tts/tts_client.py)
-------------------------------------
• Wraps ElevenLabs SDK (>=1.0).  Model: eleven_multilingual_v2
  by default; overridable via TTSConfig.model_id.

• Voice selection: DEFAULT_VOICE_MAP maps ISO-639-1 codes to
  ElevenLabs voice IDs.  Each can be overridden via env var:
    ELEVENLABS_VOICE_AR=<id>   (Arabic)
    ELEVENLABS_VOICE_EN=<id>   (English)
    …

• Disk cache: audio files are stored as:
    <cache_dir>/<SHA-256(voice::text)[:32]>.mp3
  A cache hit skips the API call entirely.

• Retry strategy (tenacity):
    – Retries on ConnectionError / TimeoutError / OSError
    – wait_exponential(min=2s, max=60s)
    – stop_after_attempt(5)
    – 429 / 500-503 HTTP errors are mapped to ConnectionError
      inside _retry_call so tenacity handles them uniformly.

• StubTTSClient: subclass that returns a hard-coded silent
  MP3 without hitting the API.  Used in tests and when no
  ELEVENLABS_API_KEY is set (auto-detected by PipelineConfig).

3.2 Arabic / RTL Handling
--------------------------
• is_rtl(lang_code) checks membership in a set of known RTL
  language codes: {ar, he, fa, ur, yi, dv, ps}.

• NarrationSegment.rtl is set automatically by sync_segment()
  so downstream renderers can apply bidi text direction.

• The TTS model (eleven_multilingual_v2) natively handles
  Arabic Unicode; no transliteration is needed.

• The Arabic demo segment ("seg_02") uses text:
  "هذه الدالة تحسب متتالية فيبوناتشي بشكل تكراري."
  RTL flag is verified by TestTimelineSync.test_segment_rtl_flag.

4. PHASE 3 — TIMELINE STRETCH & ALIGNMENT
────────────────────────────────────────────────────────────

4.1 Audio Duration Measurement (tts/audio_utils.py)
------------------------------------------------------
get_audio_duration(path) tries four backends in order:

  1. ffprobe (subprocess JSON): most accurate, handles any
     container / codec, reads from stream.duration.
  2. mutagen (pure Python): reads metadata tags; fast, no
     subprocess, supports MP3/FLAC/OGG/M4A/…
  3. pydub (AudioSegment.from_file): decodes the file,
     measures len(seg)/1000.  Requires ffmpeg or avconv.
  4. Raw MPEG frame scan (_duration_mp3_frames):
     scans bytes for MPEG-1 Layer-3 sync words and sums
     1152/sample_rate per frame.  Works with no external
     deps but is approximate for VBR files.

  If all backends fail, RuntimeError is raised.
  TimelineSyncer.sync_segment() catches RuntimeError and
  falls back to the segment's original_duration_estimate
  so the pipeline keeps running in degraded-graceful mode.

4.2 Stretch Computation (tts/audio_utils.py)
----------------------------------------------
compute_stretch_factor(actual, target):
  factor = target / actual
  Clamped to [0.5, 2.0] to prevent unrealistic stretch.

adjust_timestamps(timestamps, factor, offset):
  [round(t * factor + offset, 4) for t in timestamps]

4.3 TimelineSyncer (tts/timeline_sync.py)
-------------------------------------------
sync_segment(segment):
  1. Measures real audio duration.
  2. Computes original_end = max(event.timestamp + event.duration).
  3. factor = stretch(original_end → measured).
  4. Rebuilds events list with scaled timestamps and durations.
  5. Sets segment.rtl, segment.audio_duration, segment.stretch_factor.

build_master_timeline(segments):
  1. Calls sync_segment() for each segment in order.
  2. Assigns cumulative start_offset:
       cursor += audio_duration + inter_segment_gap (0.3 s default)
  3. Returns MasterTimeline(segments, total_duration, validated=True).

save(master):
  Writes two output files:
    output/master_timeline.json  — full Pydantic model dump
    output/segment_timings.json  — compact per-segment summary

4.4 Timeline Accuracy Guarantee
---------------------------------
The stretch factor is computed from the VALIDATED code's
narration, not the pre-correction draft.  In the pipeline:

  Phase 1 → healing.final_code  (validated)
  Phase 2 → TTS synthesizes narration for final_code
  Phase 3 → measures audio for that narration → stretches

Therefore typing/highlight events always reflect the code
that actually runs successfully.

8. KNOWN LIMITATIONS & FUTURE WORK
────────────────────────────────────────────────────────────

• Docker isolation is off by default; enable via
  SandboxConfig(use_docker=True) and ensure docker daemon
  is running.

• Stretch factor is clamped to [0.5, 2.0]; segments with
  very fast TTS relative to event density may need manual
  pacing adjustments.

• The self-correction LLM sees at most the last 3 prior
  attempts; very complex multi-error fixes may need more
  context (raise history window or use a longer-context model).

• VBR MP3 duration via frame scanning is approximate;
  install ffprobe for accuracy in production.

• ElevenLabs voice IDs may change; keep DEFAULT_VOICE_MAP
  in tts/tts_client.py up to date.
